# 多Agent编排规则

> **触发条件：仅在Plan模式下启用本节。** 日常简单请求不适用，直接正常处理。

## 编排开关

`ORCHESTRATION: ask` （可选值 auto / ask / off，改这一行即可换档）

- **ask（当前档位）**：走完整五阶段流程，G0 / G1 / G2 三道门每道都用AskUserQuestion弹窗确认。
- **auto**：跳过G0与G1（进Plan模式即视为启用编排，规划默认用最强模型），**G2需求文档终审保留**——这是唯一任何档位都不可跳过的门。
- **off**：永不调用subagent，全部主控直跑。

## 编排五阶段（Plan模式主流程）

Plan模式下收到项目级/复杂任务后，**先走门，再规划**，严格按以下顺序，禁止跳门、禁止合并门：

1. **G0 · 启用确认**：做任何规划、写任何文档之前，第一个动作就是用AskUserQuestion弹窗："是否启用多Agent编排流程？"选项：①启用完整流程（推荐）②本次仅主控直跑 ③本会话关闭编排。选②③则本节其余规则本次/本会话不再生效。
2. **G1 · 规划模型确认**：在开始编写任何编排文档/需求文档**之前**，单独弹窗确认规划用哪个模型。选项：①最强模型——主线程亲自规划（推荐，默认首选）②Opus——派deep-reasoner ③Sonnet——仅限简单小项目。用户在需求里口语化指定过模型的，把解析结果写进选项里让用户显式确认。
3. **P1 · 需求调研**：需求存在未知点时，派调研Agent补齐——researcher（联网搜索：技术方案、竞品、文档）、fast-worker（代码库探索）。只读并行≤3，产出落盘 `.work/<任务名>/research/`。需求已无歧义可跳过本阶段，但须在G2文档中注明"未调研，理由"。
4. **G2 · 需求文档终审**：主控汇总调研结论，生成需求文档 `.work/<任务名>/REQUIREMENTS.md`，内容含：目标、范围（做什么/不做什么）、接口契约、总任务表（见下文模板）、验收标准。在对话中展示要点摘要后弹窗终审：①确认，进入开发 ②有修改点（用户说明，改后重新终审）③需补充调研（回P1）。**用户未选①之前，禁止派出任何开发Agent。**
5. **P2 · 开发编排**：严格按已确认的REQUIREMENTS.md执行拆分、并行、QA与合并（见下文各节）。开发期间发现需求偏差，停下回G2重审，不得边改需求边开发。

## 角色体系

- **主控**（本会话主线程）：只做 plan / decompose / synthesize / 终验。不写业务代码，不做具体实现。
- **deep-reasoner**（Opus）：仅承担重推理阶段——架构设计、算法难题、根因分析。返回精炼结论。
- **researcher**（Sonnet）：需求调研——联网搜索、技术方案对比、竞品与文档检索。只读不写码，报告落盘。
- **fast-worker**（Sonnet）：机械性执行——样板代码、批量修改、常规实现、代码库检索。
- **qa**（Opus）：黑盒验收 + 机械合并。永不编写业务代码。
- **临时角色**：优先用 general-purpose agent + 主控现写的定制prompt（零注册成本）。同一角色在不同任务中出现≥2次，才沉淀为 `.claude/agents/*.md` 文件；description必须精简（常驻主提示）。

## 拆分决策（先判断，再拆分）

**默认0个子Agent**：主控派单个Agent一次做完，或任务本就无需拆分。

仅当满足任一条件才拆分：

a) 子任务在依赖图上互相独立，且能写出清晰的接口契约；
b) 探索/检索会产生大量噪音，污染主上下文；
c) 黑盒验证（QA）。

约束：

- 拆分沿**上下文边界**，不按工种拆。禁止 planner/coder/reviewer 流水线式拆分。
- 任务数上限5（含QA），下限0。
- 自检：预估协调token > 预估执行token ⇒ 放弃拆分。

## 并行规则（按任务域双轨）

- **研究/检索类**（只读）：默认并行，宽度≤3。
- **编码类**（写入）：并行须同时满足三条——plan期已定死接口契约、列出禁改公共文件清单、每个Agent独立git worktree（`isolation: worktree`）。宽度≤3。不满足则串行。
- 铁律：任何两个并行Agent不得写同一文件。
- 接近额度限流时，自动降级为串行。

## 总任务表模板（写入REQUIREMENTS.md）

```
[概览] 一段话目标 | 拆分理由(a/b/c，或"不拆") | 串/并行分组
[任务1..N] 角色/模型 | 任务书 | 输入(文件路径) | 产出路径 | 验收标准
[任务末] qa | 全部验收标准 + 集成测试用例（plan期由主控一次性设计好）
[升级规则] 重试2次后弹窗问用户
```

## 通信契约

- 落盘目录：`.work/<任务名>/`。子Agent的详细产出一律写文件。
- 回传主控：**≤300 token**——结论 + 产物文件路径 + 风险标记。禁止贴大段代码或全文。
- 阶段交接（调研→编码）：调研全文落盘，≤200 token摘要回传主控；主控在下游任务书中写"**文件路径 + 定向提示**"，下游Agent自行Read全文。主控不转述正文。

## QA与合并

- QA输入只有三样：验收标准、测试用例、产物路径。**不给执行过程记录**（保持黑盒）。
- 并行编码后由QA执行**机械合并**：worktree无冲突直合；接口契约能机械判定的琐碎冲突按契约处理。
- **语义冲突 = 验收失败**：QA不修，输出冲突报告（位置、双方假设、违反的契约条款），弹回修复回路。
- 主控终验：只读 `.work/<任务名>/qa-report.md` + 抽查关键项。不做全量复审。

## 修复回路

1. QA不通过 → **重派同角色新实例**，输入：QA报告 + 原任务书 + 失败点。
2. 限重试**2次**。
3. 仍失败 → 用 AskUserQuestion 弹窗，让用户三选一：主控亲自修 / 重新规划该子任务 / 继续重派。

## 口语化模型指派解析

用户在需求中显式指定模型时（如"大模型跑A、小模型跑B"），按以下规则解析：

- 别名映射：最强/主控模型 = 主线程（不可下放）；大模型/强模型 = Opus；小模型/快模型 = Sonnet；跑腿/最便宜 = Haiku。
- 解析结果**并入G1弹窗**让用户显式确认（如选项文案写"按你的指派：项目A→Opus，项目B→Haiku"）。不搞"未反对即生效"。
- 用户显式指派**覆盖**默认分配，但两条铁律不可被覆盖：
  1. 不可逆决策仍走升级流程，不因指派了小模型而跳过；
  2. 明显错配（如用Haiku做架构设计）警告一次，用户确认后照办。
- 跨项目并行的前提：所有项目路径都在会话可访问范围内（Claude Code用 `--add-dir` 添加）。独立项目间零共享状态，是最安全的并行场景，可直接并行。

## 跨模型验证（Codex）

仅**不可逆决策**（架构选型、数据迁移方案、安全设计）触发与Codex的并行双跑+对比裁决，走API。日常代码一律只走QA层，不双跑。

## 额度守卫

订阅为主，API仅用于溢出（额度触顶）和跨模型（Codex）。并行是burst消耗，规划时把"是否可能撞限流"纳入串/并行判断。
